> 创建日期: 2026-04-06
> 状态: Draft
> 作者: zhaodeezhu + Claude

## 一、背景与动机

SDD 工作流在新功能开发场景已较完善，但 Bug 修复场景存在以下缺口：

| 缺口 | 现状 | 影响 |
|------|------|------|
| Bug 修复流程 | 仅有 Team 3 "问题排查团队"模板 | 只是 team 配置，不是结构化流程 |
| Bug 文档沉淀 | 宪法中有"经验教训"章节但无结构化写入机制 | 经验散落，无法被未来复用 |
| Bug 案例库 | 不存在 | 每个 bug 从零排查，无法参考历史 |
| 与已有 spec 的关联 | 外部 bug 无关联链路 | 功能开发完后的 bug 缺少与原始 spec/plan 的关联 |
| 回归测试 | bug 修复后无回归补充机制 | 同一 bug 可能反复出现 |

### 核心目标

1. **快速定位根因** — 3 Agent 并行排查，多角度分析
2. **防止同类 bug 再犯** — 提炼防坑规则到宪法，形成案例库
3. **完整文档沉淀** — 自动生成 bug 报告、补充回归测试
4. **融入 SDD 闭环** — 修复后反馈到 spec/plan/testcases/constitution

### 适用场景

- 主要：开发自测阶段发现的 bug
- 输入：关联已有 feature_id + bug 描述（现象、复现步骤、预期行为）

### 非适用场景

- 线上紧急热修复（应走独立 hotfix 流程）
- 探索性原型开发中的问题（直接调试即可）
- 单文件 typo 修复（无需结构化流程）

## 二、整体架构

### 输入格式

```
/sdd-fix {feature_id} "{bug描述}"
/sdd-fix {feature_id} --bug-file {path}     # 从文件读取 bug 描述
/sdd-fix {feature_id} --from investigate    # 从排查阶段继续
```

- `feature_id`: 关联的已有功能编号（如 001）
- Bug 描述包含：现象、复现步骤、预期行为

### 输出产物（全部保存到已有 spec 目录）

```
.specify/specs/{feature_id}/
├── ...（已有文件）
├── bugs/                           ← 新增 bug 目录
│   ├── BUG-001-{slug}.md          ← Bug 报告（根因分析+修复记录+回归测试）
│   └── BUG-002-{slug}.md
├── spec.md                         ← 如有规格遗漏会更新
├── testcases.md                    ← 追加回归测试用例
└── plan.md                         ← 如有设计缺陷会更新
```

### 全流程架构

```
/sdd-fix 001 "提交表单后数据丢失"
         │
         ▼
┌──────────────────────────────────────────────┐
│  Phase 0: Bug 接收与预分析                      │
│    解析描述 → 加载上下文（渐进式）                 │
│    → 确认 bug 严重度与影响范围                    │
│         │                                     │
│         ▼                                     │
│  Phase 1: 根因分析（3 Agent 并行排查）           │
│    ├─ A: 代码侦探 → 代码逻辑追踪                 │
│    ├─ B: 规格对比 → 对照 spec/plan 找偏差        │
│    └─ C: 测试侦探 → 分析测试盲区                 │
│    → 仲裁合并 → 输出根因报告                     │
│         │                                     │
│         ▼                                     │
│  Phase 2: 修复实施                              │
│    写回归测试（🔴红）→ 修复代码（🟢绿）→ 验证（🔵重构）│
│         │                                     │
│         ▼                                     │
│  Phase 3: 沉淀闭环                              │
│    ├─ 追加回归测试到 testcases.md                │
│    ├─ 更新 spec/plan（如需）                     │
│    ├─ 生成 BUG-xxx.md 报告                      │
│    └─ 提炼经验到 constitution.md                │
│         │                                     │
│         ▼                                     │
│  输出 Bug 修复报告 → 人类验证                    │
└──────────────────────────────────────────────┘
```

## 三、渐进式上下文加载

sdd-fix 遵循"按需加载"原则，不一次性加载所有文档到上下文：

```
Phase 0: Bug 接收
  └── 只加载: constitution.md（查严重度阈值）+ spec 目录结构（确认关联）

Phase 1: 根因分析
  └── 每个 Agent 只加载自己需要的上下文（见 Phase 1 详述）

Phase 2: 修复实施
  └── 只加载根因相关的代码文件 + 回归测试文件

Phase 3: 沉淀
  └── 按需更新各文档，不一次性读取全部
```

Bug 报告独立存储在 `bugs/` 目录下，每个 bug 一个文件。新功能开发时不需要加载所有历史 bug。

## 四、Phase 0: Bug 接收与预分析

### 执行步骤

1. **解析输入**：提取 feature_id、bug 描述
2. **确认关联**：检查 `.specify/specs/{feature_id}/` 目录是否存在
3. **加载最小上下文**：
   - constitution.md（了解项目约束）
   - spec 目录结构（了解有哪些文档）
4. **预分析 bug**：
   - 根据描述推断可能涉及的模块/文件
   - 评估严重度（HIGH/MEDIUM/LOW）
   - 评估影响范围（单模块/跨模块/全系统）
5. **进入全自动模式**：展示预分析结果后直接进入全自动（无需人类额外确认，与 sdd-run 行为一致）

### 预分析输出

```
━━━ Bug 预分析 ━━━

关联功能: {feature_id} - {feature_name}
严重度: {HIGH/MEDIUM/LOW}
影响范围: {单模块/跨模块/全系统}
可能涉及: {推测的模块/文件}

🚀 开始全自动排查...
```

## 五、Phase 1: 根因分析（3 Agent 并行排查）

### 设计理念

借鉴 MACE 评审的多 Agent 竞争模式，从三个不同角度并行排查同一个 bug：
- 独立性：每个 Agent 有独立上下文窗口
- 分工性：从不同维度排查，减少盲区
- 仲裁性：通过共识机制提高根因分析准确性

### 3 个排查 Agent

| Agent | 角色 | 排查方向 | 输入上下文 |
|-------|------|----------|------------|
| A: 代码侦探 | 从代码逻辑追踪 bug | 逐层追踪代码执行路径，定位出错的具体代码行 | plan.md + 相关代码文件 |
| B: 规格对比 | 对照文档找偏差 | 检查实现是否偏离 spec/plan 的定义，找出遗漏或误解 | spec.md + 相关代码文件 |
| C: 测试侦探 | 分析测试盲区 | 检查该场景为何没被测试覆盖，找到测试用例的盲区 | testcases.md + 相关代码文件 |

### Agent A: 代码侦探

**核心任务**：从 bug 现象出发，追踪代码执行路径，定位出错的具体代码。

Prompt 要点：
- 从 bug 描述的现象出发，追踪可能的数据流路径
- 逐层检查：输入验证 → 业务逻辑 → 数据存取 → 响应构造 → UI 渲染
- 重点关注：空值、并发、类型不匹配、边界越界、异步时序
- 输出：可能的根因列表（按可能性排序），每个根因标注代码定位

### Agent B: 规格对比

**核心任务**：对照 spec/plan 检查实现是否偏离设计。

Prompt 要点：
- 对照 spec.md 中的用户故事，检查 bug 涉及的场景是否有定义
- 对照 plan.md 中的技术方案，检查实现是否与设计一致
- 区分"规格遗漏"（spec 没定义）和"实现偏离"（spec 有定义但实现不一致）
- 输出：偏差列表，每个偏差标注关联的文档条款和代码位置

### Agent C: 测试侦探

**核心任务**：分析为什么这个 bug 没被测试捕获。

Prompt 要点：
- 检查 testcases.md 中是否缺少覆盖该场景的测试用例
- 分析已有测试用例的覆盖盲区
- 输出：测试盲区分析 + 建议补充的回归测试场景

### 仲裁规则

收集 3 个 Agent 的排查结果后执行仲裁：

1. **根因确认**：
   - ≥2 个 Agent 找到同一根因 → **确认根因**（高置信度）
   - 仅 1 个 Agent 找到的根因 → **候选根因**，标记发现者

2. **根因分类**（决定后续修复策略）：
   - **编码错误**：代码逻辑写错了 → L1 修复（直接改代码）
   - **规格遗漏**：spec/plan 没覆盖该场景 → L2 修复（先补充文档再改代码）
   - **设计缺陷**：plan 的方案本身有问题 → L2/L3 修复（可能需要重新设计）

3. **置信度评估**：
   - 高：≥2 个 Agent 确认的根因
   - 中：1 个 Agent 找到 + 另一个 Agent 的结果间接支持
   - 低：仅 1 个 Agent 发现，其他 Agent 未涉及

### 仲裁输出格式

```
━━━ 根因分析报告 ━━━

确认根因: {根因描述}
置信度: {高/中/低}
根因分类: {编码错误/规格遗漏/设计缺陷}
代码定位: {文件:行号}
关联文档: {spec.md#章节 或 plan.md#章节}

Agent 排查详情:
  A-代码侦探: {发现摘要}
  B-规格对比: {发现摘要}
  C-测试侦探: {发现摘要}

测试盲区: {该场景未被覆盖的原因}
建议回归测试: {测试场景描述}
```

## 六、Phase 2: 修复实施（红-绿-重构）

### 设计理念

与 sdd-implement 一脉相承的红-绿-重构流程，但针对 bug 修复场景做了调整：
- 先让 bug 可复现（回归测试），再修复
- 做最小修改，不引入额外变更
- 修复后验证不引入回归

### 执行步骤

#### 1. 红阶段 - 写回归测试

- 基于 Phase 1 的根因分析，编写一个能复现 bug 的测试用例
- 运行确认测试失败（证明 bug 存在）
- 测试用例标记为回归测试（RT-{id}），关联 BUG-{seq}

#### 2. 绿阶段 - 修复 bug

- 根据根因分类执行不同策略：
  - **编码错误**：直接修改出错代码
  - **规格遗漏**：先补充 spec.md 中遗漏的场景 → 修正 plan.md → 再修复代码
  - **设计缺陷**：先修正 plan.md 的设计 → 再修复代码
- 做最小修改使回归测试通过
- 记录 CR（变更记录）

#### 3. 重构阶段 - 验证修复

- 运行刚写的回归测试确认通过
- 运行相关的其他测试确认未引入回归
- 如有完整测试套件，运行确认全部通过

### 反馈机制

修复过程中如发现新的问题，按 L1/L2/L3 级别处理（与 sdd-implement 一致）。

## 七、Phase 3: 沉淀闭环

### 设计理念

这是 sdd-fix 区别于"修完就完事"的核心环节。每次 bug 修复都必须完成四项沉淀工作，形成"开发→发现 bug→修复→沉淀→防止再犯"的良性循环。

### 7.1 追加回归测试到 testcases.md

在现有 testcases.md 末尾新增"回归测试"章节：

```markdown
## 回归测试

### BUG-001: {bug 标题}

> 修复日期: {date}
> 根因类型: {编码错误/规格遗漏/设计缺陷}

#### RT-{id}: {测试名称}
- **类型**: {UT/IT/E2E}
- **Given**: {触发条件}
- **When**: {操作}
- **Then**: {预期结果}
- **关联Bug**: BUG-001
```

### 7.2 更新 spec/plan（如有遗漏）

根据根因分类，更新上游文档：

- **规格遗漏类** → 在 spec.md 中补充遗漏的边界条件/异常场景
- **设计缺陷类** → 在 plan.md 中补充遗漏的处理逻辑
- 使用 CR 记录追踪变更

### 7.3 生成 Bug 报告

保存到 `.specify/specs/{feature_id}/bugs/BUG-{seq}-{slug}.md`：

```markdown
# Bug 报告: BUG-001

> 功能: {feature_id} - {feature_name}
> 发现日期: {date}
> 修复日期: {date}
> 根因类型: {编码错误/规格遗漏/设计缺陷}
> 严重度: {HIGH/MEDIUM/LOW}

## 现象描述

{bug 现象和复现步骤}

## 根因分析

### 排查方法
3 Agent 并行排查（代码侦探 + 规格对比 + 测试侦探）

### 确认根因
{最终确认的根因，含代码定位}

### 根因分类
- [ ] 编码错误：代码逻辑写错
- [ ] 规格遗漏：spec/plan 未覆盖该场景
- [ ] 设计缺陷：plan 的方案本身有问题

## 修复方案

{修复了什么，怎么修的}

### 变更文件
- {文件路径}: {变更说明}

## 回归测试

- RT-{id}: {测试名称} → 已添加到 testcases.md

## 防坑规则

> {提炼的经验教训，一句话规则}
> 已沉淀到: constitution.md#经验教训

## 关联文档

- 功能规格: spec.md#{相关章节}
- 技术方案: plan.md#{相关章节}
- 测试用例: testcases.md#RT-{id}

## CR 记录

| CR编号 | 变更摘要 | 级别 |
|--------|---------|------|
| CR-BUG-{seq}-1 | {摘要} | {级别} |
```

### 7.4 提炼经验到 constitution.md

在 constitution.md 的"经验教训"章节追加（增量更新，不重写整个文件）：

```markdown
### BUG-{seq}: {一句话描述问题}
- **规则**: {防坑规则，未来开发时必须遵守}
- **根因**: {根因一句话}
- **修复**: {修复方式一句话}
- **参考**: bugs/BUG-{seq}-{slug}.md
```

### 良性循环

```
开发功能 → review 通过 → 测试发现 bug
    ↑                          │
    │                          ▼
    │               /sdd-fix 排查根因
    │                          │
    │                          ▼
    │               修复 + 回归测试
    │                          │
    │                          ▼
    │               沉淀到 testcases.md（回归测试）
    │                          │
    │                          ▼
    │               沉淀到 constitution.md（防坑规则）
    │                          │
    └──────────────────────────┘
    下次开发时：
    - testcases.md 有更多覆盖（减少遗漏）
    - constitution.md 有更多防坑规则（避免重犯）
    - bugs/ 目录有历史案例可参考（快速定位）
```

## 九、与 SDD 流程的关系

### 与现有 Skill 的关系

| 现有 Skill | sdd-fix 的关系 | 变更 |
|-----------|---------------|------|
| sdd-specify | fix 发现规格遗漏时更新 spec | sdd-fix 不修改 sdd-specify |
| sdd-testcases | fix 追加回归测试到 testcases | sdd-fix 不修改 sdd-testcases |
| sdd-plan | fix 发现设计缺陷时更新 plan | sdd-fix 不修改 sdd-plan |
| sdd-implement | 修复代码时遵循相同红-绿-重构流程 | sdd-fix 不修改 sdd-implement |
| sdd-review | review 发现的 bug 可转交 fix 处理 | sdd-fix 不修改 sdd-review |
| sdd-run | fix 是独立 skill，不嵌入 sdd-run | 未来可扩展为 sdd-run --mode fix |
| sdd-constitution | fix 增量更新宪法的经验教训 | sdd-fix 不修改 sdd-constitution |

### sdd-fix 是独立 Skill

- 有自己的 SKILL.md 和 manifest.json
- 不修改任何现有 skill
- 通过文件系统与其他 skill 协作（读写相同的 spec 目录和文档）

### 反馈级别与 sdd-implement 一致

| 级别 | 场景 | 处理方式 |
|------|------|----------|
| L1 | 编码错误，直接修代码 | 自动修复 |
| L2 | 规格遗漏/设计缺陷 | 自动补充文档 + 修复代码 |
| L3 | 修复过程发现方向性问题 | 暂停，等人类决策 |

## 十一、设计原则总结

1. **与 SDD 理念一致** — 文档驱动、多 Agent、红-绿-重构、CR 记录
2. **渐进式披露** — 按需加载上下文，不一次性加载所有文档
3. **增量沉淀** — 不重写已有文档，只追加回归测试和防坑规则
4. **独立 skill** — 不修改现有 skill，通过文件系统协作
5. **良性循环** — 修复 → 沉淀 → 未来开发参考 → 避免重犯
6. **全自动模式** — 人类只需提供 bug 描述和验证最终结果
