# SDD Fix - Bug 修复器（3 Agent 并行排查 + 沉淀闭环）

> **与 MACE 评审一脉相承**：借鉴多 Agent 竞争评审的独立性和仲裁机制，
> 从代码、规格、测试三个角度并行排查同一个 bug，通过共识机制确认根因。

全自动 Bug 修复流程：接收 bug 描述 → 3 Agent 并行排查根因 → 红-绿-重构修复 → 文档沉淀闭环。
人类只需提供 bug 描述和验证最终结果。

## 核心定位

> Fix 是 SDD 流程的**Bug 修复专用 skill**，与功能开发流水线互补。
> 修复后的经验自动沉淀到项目宪法和测试用例，形成"修一次防一类"的良性循环。

## 前置条件

- 已有功能目录: `.specify/specs/{feature_id}/`
- 已有功能规格: `.specify/specs/{feature_id}/spec.md` 或 `spec/README.md`
- 项目宪法: `.specify/memory/constitution.md`

## 输入格式

```
/sdd-fix {feature_id} "{bug描述}"
/sdd-fix {feature_id} --bug-file {path}     # 从文件读取 bug 描述
/sdd-fix {feature_id} --from investigate    # 从排查阶段继续
```

- `feature_id`: 关联的已有功能编号（如 001）
- Bug 描述包含：现象、复现步骤、预期行为

### 总体架构

```

│    解析描述 → 加载上下文（渐进式）                 │

│         │                                     │
│         ▼                                     │
│  Phase 1: 根因分析（3 Agent 并行排查）           │
│    ├─ A: 代码侦探 → 代码逻辑追踪                 │
│    ├─ B: 规格对比 → 对照 spec/plan 找偏差        │
│    └─ C: 测试侦探 → 分析测试盲区                 │
│    → 仲裁合并 → 输出根因报告                     │

│         ▼                                     │
│  Phase 2: 修复实施                              │
│    写回归测试（🔴红）→ 修复代码（🟢绿）→ 验证（🔵重构）│

│    ├─ 追加回归测试到 testcases.md                │

│         │                                     │

└──────────────────────────────────────────────┘
```

### Phase 0: Bug 接收与预分析

> **渐进式加载**：此阶段只加载 constitution.md + spec 目录结构，不加载全部文档。

**执行步骤**：

#### 0.1 解析输入

1. 提取 `feature_id` 和 bug 描述
2. 如使用 `--bug-file`，使用 Read 工具读取指定文件获取 bug 描述
3. 如使用 `--from`，跳到指定阶段

#### 0.2 确认关联

1. 检查 `.specify/specs/{feature_id}/` 目录是否存在
2. 如不存在，提示用户先使用 `/sdd-specify` 创建功能规格
3. 列出该目录下已有的文档文件

#### 0.3 加载最小上下文

只加载以下内容（渐进式，不加载全部）：

1. `.specify/memory/constitution.md` — 项目约束
2. `.specify/specs/{feature_id}/` 目录结构 — 了解有哪些文档
3. **不加载** spec.md、plan.md、testcases.md 的完整内容（留给 Phase 1 的 Agent）

#### 0.4 预分析 bug

1. 根据描述推断可能涉及的模块/文件
2. 评估严重度（HIGH/MEDIUM/LOW）
3. 评估影响范围（单模块/跨模块/全系统）
4. 确定 bug 编号：扫描 `bugs/` 子目录已有文件，取最大编号 + 1

#### 0.5 展示预分析结果并进入全自动

```
━━━ Bug 预分析 ━━━

关联功能: {feature_id} - {feature_name}
严重度: {HIGH/MEDIUM/LOW}
影响范围: {单模块/跨模块/全系统}
可能涉及: {推测的模块/文件}

🚀 开始全自动排查...
```

展示后直接进入 Phase 1，无需等待人类确认。

### Phase 1: 根因分析（3 Agent 并行排查）

> **核心设计**：借鉴 MACE 评审的多 Agent 竞争模式。
> 3 个 Agent 在**同一条消息中并行派发**，每个 Agent 有独立上下文窗口。
> 每个 Agent 只加载自己需要的上下文（渐进式披露）。

#### 1.1 确定代码范围

在派发 Agent 之前，先确定需要排查的代码文件范围：

1. 读取 `.specify/specs/{feature_id}/plan.md` 或 `plan/README.md`，提取涉及的文件路径
2. 如果 plan.md 未列出具体文件路径，根据 plan 中的模块描述和项目结构推断
3. 记录所有需要排查的代码文件路径

#### 1.2 并行派发 3 个排查 Agent

使用 Agent 工具**同时派发 3 个独立排查 Agent**。

##### Agent A: 代码侦探

派发参数：
- **subagent_type**: `general-purpose`
- **description**: `SDD排查-代码侦探`
- **prompt** 内容模板：

```

## 技术方案

{此处注入 plan.md 的完整内容}

## 待排查代码文件

{代码文件路径列表，每行一个}

## 排查任务

从 bug 现象出发，追踪可能的代码执行路径：

1. **数据流追踪**：从用户输入到最终输出，逐层检查数据处理：
   - 输入验证：参数是否正确接收和校验？
   - 业务逻辑：处理流程是否符合预期？
   - 数据存取：数据库操作是否正确？
   - 响应构造：返回数据格式是否正确？
   - 前端渲染：UI 展示逻辑是否正确？

2. **重点关注**：
   - 空值/null 处理
   - 并发/竞态条件
   - 类型不匹配
   - 边界越界
   - 异步时序问题
   - 数据转换错误

3. **输出要求**：列出所有可能的根因，按可能性从高到低排序。

## 输出格式

严格按以下格式输出，不要输出其他内容：

- [{编码错误|规格遗漏|设计缺陷}] {分类理由}
SUPPORTING_EVIDENCE:
- {支持证据：具体的代码片段或逻辑分析}
---INVESTIGATE-END---

#### 1.3 仲裁合并

收集 3 个 Agent 的排查结果后，执行仲裁：

##### 根因确认

1. 解析每个 Agent 的输出，提取根因列表
2. 比对 3 份结果，按语义相似度分组
3. 确认规则：
   - **≥2 个 Agent 找到同一根因** → **确认根因**（高置信度）
   - **仅 1 个 Agent 找到的根因** → **候选根因**（低置信度，但保留参考）

##### 根因分类

根据仲裁结果，将根因分为三类：

| 分类 | 含义 | 修复策略 |
|------|------|----------|
| 编码错误 | 代码逻辑写错了 | L1：直接修改代码 |
| 规格遗漏 | spec/plan 没覆盖该场景 | L2：先补充 spec/plan，再修改代码 |
| 设计缺陷 | plan 的方案本身有问题 | L2/L3：修正 plan，可能需要人类决策 |

##### 置信度评估

- **高**：≥2 个 Agent 确认的根因
- **中**：1 个 Agent 找到 + 另一个 Agent 的结果间接支持
- **低**：仅 1 个 Agent 发现，其他 Agent 未涉及

##### 仲裁输出

```
━━━ 根因分析报告 ━━━

确认根因: {根因描述}
置信度: {高/中/低}
根因分类: {编码错误/规格遗漏/设计缺陷}
代码定位: {文件:行号}
关联文档: {spec.md#章节 或 plan.md#章节}

Agent 排查详情:
  A-代码侦探: {发现摘要}
  B-规格对比: {发现摘要}
  C-测试侦探: {发现摘要}

测试盲区: {该场景未被覆盖的原因}
建议回归测试: {测试场景描述}
```

仲裁完成后自动进入 Phase 2。

---

### Phase 2: 修复实施（红-绿-重构）

> **与 sdd-implement 一脉相承**，但专注于 bug 修复：先复现，再修复，不引入额外变更。

#### 2.1 🔴 红阶段 - 写回归测试

1. 基于 Phase 1 的根因分析，编写一个能复现 bug 的测试用例
2. 测试用例标记为回归测试（RT-{id}），关联 BUG-{seq}
3. 运行确认测试失败（证明 bug 存在）
4. 如测试未失败，说明 bug 未能复现，需要回到 Phase 1 重新排查

#### 2.2 🟢 绿阶段 - 修复 bug

根据根因分类执行不同修复策略：

**编码错误（L1）**：
1. 直接修改出错的代码
2. 做最小修改，不引入额外变更
3. 记录 CR（CR-BUG-{seq}-1）

**规格遗漏（L2）**：
1. 在 spec.md 中补充遗漏的边界条件/异常场景
2. 在 plan.md 中补充对应的技术处理方案
3. 修改代码实现补充的处理方案
4. 记录 CR（CR-BUG-{seq}-1）

**设计缺陷（L2/L3）**：
1. 在 plan.md 中修正设计方案
2. 如修正影响范围大（L3），暂停等待人类决策
3. 修改代码实现修正后的方案
4. 记录 CR（CR-BUG-{seq}-1）

#### 2.3 🔵 重构阶段 - 验证修复

1. 运行回归测试确认通过
2. 运行相关的其他测试确认未引入回归
3. 如有完整测试套件，运行确认全部通过

#### 2.4 反馈处理

修复过程中如发现新问题：

| 发现的问题 | 反馈级别 | 处理方式 |
|-----------|---------|---------|
| 修复引入新的小问题 | L1 | 自动修正，记录 CR |
| 发现 spec/plan 有其他遗漏 | L2 | 自动补充，级联更新 |
| 修复方向有根本性问题 | L3 | 暂停，输出影响分析，等人类决策 |

### Phase 3: 沉淀闭环

> **核心价值**：修完不是终点，沉淀才是。每次 bug 修复必须完成四项沉淀工作。

#### 3.1 追加回归测试到 testcases.md

在 testcases.md 末尾追加回归测试章节：

```markdown
## 回归测试

> 修复日期: {date}

#### RT-{id}: {测试名称}

- **Given**: {触发条件}
- **When**: {操作}
- **Then**: {预期结果}
- **关联Bug**: BUG-{seq}
```

#### 3.2 更新 spec/plan（如有遗漏）

根据根因分类决定是否更新上游文档：

- **规格遗漏类** → 在 spec.md 中补充遗漏的边界条件/异常场景，记录 CR
- **设计缺陷类** → 在 plan.md 中补充遗漏的处理逻辑，记录 CR
- **编码错误类** → 不更新上游文档

#### 3.3 生成 Bug 报告

使用 Write 工具将 Bug 报告保存到 `.specify/specs/{feature_id}/bugs/BUG-{seq}-{slug}.md`。

#### 3.4 提炼经验到 constitution.md

```markdown

```

### `--from` 参数

```
/sdd-fix 001 --from investigate    # 跳过 Phase 0，从排查开始

```

### L3 暂停恢复

- 阻塞原因和影响分析
- 建议的处理方案
- 当前已完成阶段和产出文件

## 与其他 SDD Skill 的关系

| 现有 Skill | sdd-fix 的关系 |
|-----------|---------------|
| sdd-specify | fix 发现规格遗漏时更新 spec |
| sdd-testcases | fix 追加回归测试到 testcases |
| sdd-plan | fix 发现设计缺陷时更新 plan |
| sdd-implement | 修复代码时遵循相同红-绿-重构流程 |
| sdd-review | review 发现的 bug 可转交 fix 处理 |
| sdd-constitution | fix 增量更新宪法的经验教训 |

## 注意事项

1. **渐进式加载**：每个 Phase 只加载必要的上下文，不一次性加载所有文档
2. **并行派发**：3 个排查 Agent 必须在同一条消息中并行派发，不能串行执行
3. **独立性**：每个排查 Agent 拥有独立上下文窗口，文档内容直接注入 Agent prompt
4. **增量更新**：testcases.md 和 constitution.md 使用增量追加，不重写整个文件
5. **CR 记录**：所有文档变更都记录 CR，保证可追溯性
6. **防坑规则**：每次修复必须提炼一条可操作的防坑规则，写入 constitution.md
7. **Bug 报告必须保存**：使用 Write 工具保存 Bug 报告，保存后使用 Read 确认完整性
8. **不自动提交**：代码修改不自动 git commit，由人类验收后决定
9. **L2+ 级别的反馈经验**：必须提炼为防坑规则追加到 constitution.md

## Bug 修复流程（sdd-fix）

### 问题

SDD 的功能开发流水线很完善，但开发完后的 Bug 修复是割裂的：
- Bug 修完没有文档记录，经验无法复用

### 解决方案：sdd-fix

`sdd-fix` 是一个全自动 Bug 修复 skill，核心创新是**3 Agent 并行排查 + 沉淀闭环**：

```
/sdd-fix 001 "提交表单后数据丢失"
     │
     ▼
Phase 1: 根因分析
  ├─ A: 代码侦探（追踪代码逻辑）
  ├─ B: 规格对比（对照 spec/plan 找偏差）

  ├─ 回归测试追加到 testcases.md
  ├─ Bug 报告保存到 bugs/ 目录
  └─ 防坑规则提炼到 constitution.md
```
